# Thinking/Reasoning 处理机制

## 1. 设计目标：零空泡 (Zero Empty Bubbles)
在 AI 模型输出结果前，通常会先产生一段 "Thinking"（推理）内容。由于钉钉的消息展示限制，如果直接创建一张空的 AI 卡片来显示推理过程，会导致用户界面出现短暂的空白或不完整的卡片。

我们的设计目标是：**在推理阶段保持安静，仅在真正开始输出答案时才显示卡片。**

## 2. 延迟创建 (Lazy Creation) 模式
在 `CreateStream` 时，我们会根据流的性质决定是否采用延迟创建：

- **First Stream (Reasoning Lane)**:
  - 标记为 `isLazy=true`。
  - 收到 `_Reasoning:_` 开头的内容时，仅更新内部 buffer，不调用钉钉 API 创建卡片。
  - 如果模型在停止前只输出了推理内容（例如某些纯过滤任务），则整个流结束时不会在钉钉产生任何视觉反馈。
- **Answer Stream (Answer Lane)**:
  - 当模型从推理转入正文，或直接输出正文时。
  - 在 `flush()` 中检测到非推理内容，立即触发 `CreateAICard`。
  - 此时卡片才会出现在用户的聊天窗口中。

## 3. 情绪贴纸 (Thinking Sticker) 交互
为了弥补延迟创建卡片带来的反馈缺失，我们利用了钉钉的 "Emotion Reply" 接口：

1. **收到消息时**：立即在用户消息下方添加一个 "思考中" 的表情图标。
2. **推理/生成中**：用户看到表情图标，知道 AI 正在处理。
3. **输出完毕/流停止时**：调用 `RecallEmotionReply` 撤回该表情，同时最终完成 AI 卡片的渲染。

这种组合方案既能提供即时的处理反馈，又保证了最终回复内容的整洁度。
