# 钉钉 AI 卡片机制与生命周期

## 1. 卡片类型
GoClaw 钉钉频道使用 **AI 交互卡片** (Interactive Card) 作为主要交互载体，而不是普通的文本消息。

- **优势**：支持实时流式更新 (Streaming)、更丰富的 Markdown 渲染、内置按钮交互、以及更好的视觉呈现（头部状态标识）。
- **生命周期状态**：
  - `1`: 初始状态（不常用）
  - `2`: 处理中 (Processing) - 头部显示旋转动画，底色为蓝色。
  - `3`: 已完成 (Finished) - 头部固定，底色恢复正常。

## 2. 映射与幂等性管理
由于 DingTalk Stream 可能会因重连导致消息重复推送，或者在流式生成与最终下发之间存在竞态，我们维护了两个核心 map：

- **`msgIDToCardID` (消息 ID -> 卡片 TrackID)**：
  - 记录某条用户消息对应的响应卡片 ID。
  - 确保即使在流停止后，通过 `Send()` 下发的最终完整回复也能准确编辑到同一张卡片上，避免产生两条回复。
- **`deliveredMsgs` (已投递消息记录)**：
  - 当流式输出已经完整渲染了最终答案，会将该消息 ID 标记为已投递。
  - 后续的 `Send()` 调用会检测该标志，若已投递则直接跳过，防止由于网络重试导致的重复回复。

## 3. 内存泄漏防护 (TTL 清理)
为了防止这两个 Map 在长运行服务中无限增长，系统内置了清理协程：
- **清理周期**：每 5 分钟执行一次。
- **过期策略 (TTL)**：清理超过 **10 分钟** 之前的映射记录。
- 由于一次对话通常在几秒到一分钟内完成，10 分钟的 TTL 既保证了交互一致性，又防止了内存堆积。

## 4. 降级方案
如果由于某些原因（如权限不足或接口故障）无法创建 AI 卡片，系统会尝试降级为普通的 `sessionWebhook` 文本回复。
